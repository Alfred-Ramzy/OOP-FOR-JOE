Classes & Objects
The foundation of OOP — Class = blueprint, Object = instance built from that blueprint
What is a Class?

A class is a template — it defines what data (attributes) and behaviour (methods) its objects will have. Like a cookie-cutter: defines the shape, but is not a cookie itself.

Analogy Class = "Car blueprint". Object = "The actual car you drive."
What is an Object?

An object is a concrete instance of a class. You can create many objects from a single class, each with its own data.

Key point p1 = Person("Ahmed") and p2 = Person("Sara") are two separate objects — different data, same class.
Full Class Anatomy — with C++ comparison
class Person:
    # ─── __init__ is the CONSTRUCTOR ────────────────────────
    #   - Runs automatically when you create an object
    #   - Equivalent to MyClass() { } in C++
    #   - 'self' is mandatory as the FIRST parameter in every method
    #   - 'self' is like 'this' pointer in C++ — refers to the object itself
    def __init__(self, name: str, age: int):
        self.name = name          # instance variable (each object owns its own copy)
        self.age  = age

    # ─── Regular Method ─────────────────────────────────────
    #   - Always receives 'self' as first param (like 'this->' in C++)
    def greet(self):
        print(f"Hello, I am {self.name} and I am {self.age} years old.")

    # ─── __str__ controls what print(object) shows ───────────
    #   - Without it: print(p1) shows: <__main__.Person object at 0x...>
    def __str__(self):
        return f"Person(name={self.name}, age={self.age})"


# ─── Creating Objects ────────────────────────────────────
#   Python: p1 = Person("Ahmed", 20)
#   C++:    Person p1("Ahmed", 20);   or   Person* p1 = new Person("Ahmed",20);

p1 = Person("Ahmed", 20)    # __init__ is called automatically
p2 = Person("Sara",  22)

p1.greet()                   # Hello, I am Ahmed and I am 20 years old.
print(p1)                    # Person(name=Ahmed, age=20)  ← uses __str__
print(p1.name)               # Ahmed

# ─── Modify attribute ────────────────────────────────────
p1.age = 25                  # direct update (like p1.age = 25 in C++)

# ─── Delete object ───────────────────────────────────────
#   Python: del p1   (garbage collected)
#   C++:    delete p1;   (if heap allocated)
del p1
# print(p1.name)  → NameError: name 'p1' is not defined
Why does every method need self?

In C++, the compiler silently passes this to every member function. Python does NOT do that — you must write self explicitly as the first parameter.

Common Mistake Forgetting self in a method definition. Python will throw a TypeError when you call it. 
class Bad:
    def greet():          # ❌ missing self
        print("hello")

class Good:
    def greet(self):      # ✅ correct
        print("hello")
Class vs Instance variables
class Student:
    school = "XYZ University"  # class variable
    # shared by ALL objects

    def __init__(self, name):
        self.name = name      # instance variable
        # unique per object

s1 = Student("Ali")
s2 = Student("Nour")

print(s1.school)  # XYZ University
print(s2.school)  # XYZ University (same)
print(s1.name)    # Ali
print(s2.name)    # Nour (different)

Encapsulation
Bundling data + methods. Controlling who can access what.
Core Idea Encapsulation = wrapping data and the methods that operate on it inside one unit (the class), and restricting direct access to some of that data. Think of it as making some variables "private" so outsiders can't accidentally break them.

Important: Python's "privacy" is not absolute Python uses Name Mangling for private vars: self.__salary is internally stored as _Employee__salary. You CAN still access it with obj._Employee__salary — it's a deterrent, not a lock.
Public & Protected — Examples
class Employee:
    def __init__(self, name, dept, salary):
        self.name   = name    # PUBLIC   — anyone can access
        self._dept  = dept    # PROTECTED — convention: subclasses only
        self.__salary = salary # PRIVATE  — blocked outside class

emp = Employee("Sara", "IT", 8000)

print(emp.name)      # ✅ Sara
print(emp._dept)     # ✅ IT  (works, but against convention)
print(emp.__salary)  # ❌ AttributeError!
# Name mangling workaround:
print(emp._Employee__salary)  # ✅ 8000 (not recommended)
Accessing Private via Method
class Student:
    def __init__(self, name, gpa):
        self.name  = name
        self.__gpa = gpa   # private

    def show(self):
        # Inside the class: private IS accessible
        print(f"{self.name}: GPA = {self.__gpa}")

s = Student("Ali", 3.9)
s.show()            # ✅ Ali: GPA = 3.9
print(s.__gpa)      # ❌ AttributeError
Getters & Setters — Controlled Access with Validation
Why use Getters/Setters? When you want to READ or WRITE a private variable but with logic — e.g., prevent a negative salary, validate an email format, or log every change. 
class BankAccount:
    def __init__(self, owner, balance):
        self.owner    = owner
        self.__balance = balance    # private — outsiders can't touch directly

    # GETTER — lets you READ the private value safely
    def get_balance(self):
        return self.__balance

    # SETTER — lets you UPDATE the private value WITH validation
    def deposit(self, amount):
        if amount > 0:
            self.__balance += amount
            print(f"Deposited {amount}. New balance: {self.__balance}")
        else:
            print("Error: deposit must be positive!")

acc = BankAccount("Ahmed", 1000)
print(acc.get_balance())    # 1000  ← getter
acc.deposit(500)            # Deposited 500. New balance: 1500
acc.deposit(-200)           # Error: deposit must be positive!
acc.__balance = 9999        # ❌ this does NOT change the real __balance
                             #    Python creates a NEW public attr instead!
Protected Members & Inheritance
class Base:
    def __init__(self):
        self._value = 42      # protected (single underscore)

class Child(Base):
    def __init__(self):
        Base.__init__(self)
        print(self._value)    # ✅ accessible in subclass: 42
        self._value = 100    # ✅ can modify it too

c = Child()           # prints 42
print(c._value)       # 100 (accessible but frowned upon outside)

Inheritance
Child class inherits attributes & methods from Parent class. The "is-a" relationship.
The Is-A Rule Use inheritance only when the relationship makes sense as "is-a": Dog IS AN Animal ✓ | Car IS A Vehicle ✓ | Car IS A Person ✗
Basic Syntax — Python vs C++
# ── Python ──────────────────────────────────────────────
#   class Child(Parent):     ← Python syntax
#
# ── C++ ─────────────────────────────────────────────────
#   class Dog : public Animal { };   ← C++ syntax
# ─────────────────────────────────────────────────────────

class Animal:
    def __init__(self, name):
        self.name = name

    def eat(self):
        print(f"{self.name} is eating.")

    def describe(self):
        print(f"I am an animal named {self.name}.")


class Dog(Animal):       # ← inherits from Animal
    def bark(self):
        print("Woof!")


d = Dog("Rex")          # uses Animal's __init__ automatically
d.eat()                  # ✅ Inherited: Rex is eating.
d.describe()             # ✅ Inherited: I am an animal named Rex.
d.bark()                 # ✅ Dog's own method: Woof!
Child with its own __init__ — Calling Parent Constructor
Critical Rule If the child defines its own __init__, it MUST explicitly call the parent's __init__ — otherwise the parent's instance variables won't exist and you'll get an AttributeError. 
class Person:
    def __init__(self, name, id_num):
        self.name   = name
        self.id_num = id_num

    def display(self):
        print(f"Name: {self.name} | ID: {self.id_num}")


class Employee(Person):
    def __init__(self, name, id_num, salary, role):
        # ─── Option 1: Call parent by name ───────────────────
        #   Person.__init__(self, name, id_num)
        #
        # ─── Option 2: super() — PREFERRED ──────────────────
        #   Doesn't require knowing the parent class name
        #   Works better with multiple inheritance
        super().__init__(name, id_num)   # ← call parent first

        self.salary = salary           # Employee's own attributes
        self.role   = role

    def show_details(self):
        self.display()                  # call inherited method
        print(f"Role: {self.role} | Salary: {self.salary}")


emp = Employee("Sara", "EMP-001", 6000, "Engineer")
emp.show_details()
# Name: Sara | ID: EMP-001
# Role: Engineer | Salary: 6000
What happens if you forget super()? 
class A:
    def __init__(self): self.x = 10

class B(A):
    def __init__(self): self.y = 20   # ← did NOT call super()

b = B()
print(b.y)   # 20 ✅
print(b.x)   # ❌ AttributeError: 'B' object has no attribute 'x'
Types of Inheritance — All 5 Types

1. Single Inheritance

Animal → Dog

One parent, one child.

class Dog(Animal): pass

2. Multiple Inheritance

Father + Mother → Child

One child, many parents.

class C(A, B): pass

3. Multilevel Inheritance

Vehicle → Car → SportsCar

Chain: grandparent → parent → child.

class SportsCar(Car): pass
# Car already inherits Vehicle

4. Hierarchical Inheritance

Animal → Dog, Cat, Horse

One parent, many children.

class Dog(Animal): pass
class Cat(Animal): pass

5. Hybrid Inheritance

Parent → Child1, Child2 → GrandChild (mix of multiple types)

Combination of two or more types of inheritance.

class A: pass
class B(A): pass    # Single
class C(A): pass    # Hierarchical
class D(B, C): pass  # Multiple → Hybrid
Method Overriding — Redefine parent behaviour in child

When a child class defines a method with the same name as one in the parent, it overrides the parent version. When you call that method on the child object, Python runs the child's version.

class Animal:
    def eat(self):
        print("I can eat.")           # generic behaviour

class Dog(Animal):
    def eat(self):                   # ← OVERRIDE: same name!
        super().eat()                # ← optional: run parent's version first
        print("I especially love bones!")  # ← then add own behaviour

dog = Dog()
dog.eat()
I can eat.
I especially love bones!
# More complete example — Bird hierarchy
class Bird:
    def intro(self): print("I am a bird.")
    def flight(self): print("Most birds can fly.")

class Sparrow(Bird):
    def flight(self): print("Sparrows can fly.")    # override

class Ostrich(Bird):
    def flight(self): print("Ostriches cannot fly.")  # override

for b in [Bird(), Sparrow(), Ostrich()]:
    b.intro()    # inherited from Bird
    b.flight()   # each calls its own overridden version
Private Members Are NOT Inherited
class Car:
    def __init__(self):
        self.color   = "Red"      # public  → inherited ✅
        self._brand  = "Toyota"   # protected → accessible in child ✅
        self.__engine = "V8"     # private → NOT inherited ❌

class BMW(Car):
    def show(self):
        print(self.color)         # ✅ Red
        print(self._brand)        # ✅ Toyota
        print(self.__engine)      # ❌ AttributeError

b = BMW()
b.show()

Polymorphism
"Poly" = many · "morphism" = forms. Same name, different behaviour.
Core Idea Python lets different objects respond to the SAME method name in their OWN way. You can treat different objects uniformly — and each will "do the right thing."
Operator Polymorphism

The + operator behaves differently based on data type — that's polymorphism!

# + on integers → addition
print(3 + 4)              # 7

# + on strings → concatenation
print("Hello" + " World")  # Hello World

# + on lists → merge
print([1,2] + [3,4])      # [1, 2, 3, 4]
Function Polymorphism — len()

len() works on strings, lists, and dicts — different logic, same function name.

print(len("Python"))          # 6  (number of chars)
print(len(["a","b","c"]))    # 3  (number of items)
print(len({"x":1, "y":2}))    # 2  (number of keys)
Class Polymorphism — Same method name, different classes
class Cat:
    def __init__(self, name, age):
        self.name = name; self.age = age
    def info(self):
        print(f"I am a cat. Name: {self.name}, Age: {self.age}")
    def sound(self): print("Meow!")

class Dog:
    def __init__(self, name, age):
        self.name = name; self.age = age
    def info(self):                           # ← same name as Cat.info
        print(f"I am a dog. Name: {self.name}, Age: {self.age}")
    def sound(self): print("Bark!")           # ← same name as Cat.sound

cat1 = Cat("Kitty", 2)
dog1 = Dog("Fluffy", 4)

# Polymorphism in action: same loop, different behaviour
for animal in (cat1, dog1):
    animal.sound()            # Meow! / Bark! — Python knows which to call
    animal.info()
Meow!
I am a cat. Name: Kitty, Age: 2
Bark!
I am a dog. Name: Fluffy, Age: 4
Method Overriding (Polymorphism + Inheritance)

When a child overrides a parent method — calling the method on different objects gives different results.

class Shape:
    def area(self):
        print("Calculating area...")

class Circle(Shape):
    def area(self):          # override
        print("Area = pi * r^2")

class Rectangle(Shape):
    def area(self):          # override
        print("Area = w * h")

for s in [Shape(), Circle(), Rectangle()]:
    s.area()
Method Overloading — Python's Way
Python has NO native overloading! Defining the same method name twice replaces the first one. Use default parameters instead. 
# ❌ This does NOT work in Python
class Bad:
    def add(self, x, y): print(x+y)
    def add(self, x, y, z): print(x+y+z)
Bad().add(2, 3)    # TypeError! (1st add is gone)

# ✅ Correct Python way: default args
def add(x, y, z=0):   # z is optional, defaults to 0
    return x + y + z

print(add(2, 3))       # 5  (z=0)
print(add(2, 3, 4))    # 9
Overriding vs Overloading — Key Difference
Method OVERRIDING
  • Same method name
  • In a child class (inheritance required)
  • Child gives its own version of the method
  • Happens at runtime
  • Python: fully supported natively
Method OVERLOADING
  • Same method name
  • In the same class
  • Different number/type of arguments
  • Happens at compile time (C++)
  • Python: NOT natively supported — use default args
